# Copilot Instructions for paas-charm

## Overview

`paas-charm` is a Python library providing base [Juju](https://juju.is/) charm classes for deploying 12-factor apps (Flask, Django, FastAPI, Go, Express, Spring Boot) on Kubernetes. It is published as a PyPI package (`paas-charm`) and consumed by per-framework charms generated by [Charmcraft](https://documentation.ubuntu.com/charmcraft/stable/) extensions.

## Build, Test, and Lint Commands

```bash
# Format code
tox -e fmt

# Lint (flake8, isort, black, mypy, pydocstyle, pylint, codespell)
tox -e lint

# Unit tests
tox -e unit

# Run a single unit test
tox -e unit -- tests/unit/flask/test_charm.py::test_flask_pebble_layer

# Security static analysis (bandit)
tox -e static

# Full suite (lint + unit + static + coverage-report)
tox
```

### Integration Tests

Integration tests require `opcli` (from `canonical/charm-ci`), Spread, and a local LXD VM. Build artifacts first:

```bash
opcli artifacts build                                          # builds charms/rocks → artifacts.build.yaml
opcli spread run -- -list                                      # list available tests
opcli spread run -- -v -debug integration-test-local:ubuntu-24.04:tests/integration/run:<test_name>
```

For faster iteration against the current host:

```bash
opcli artifacts build
opcli env provision
opcli artifacts push-images --missing-registry deploy
opcli pytest run -- -k "test_name"
```

## Architecture

### Package layout

- `src/paas_charm/` — main library; the canonical source for all new code
- `src/paas_app_charmer/` — **deprecated** thin re-export shims (maintained for Juju bases 22.04/24.04; do not add new code here)
- `examples/` — reference charm implementations for each supported framework
- `tests/unit/` / `tests/integration/` — framework-specific test suites

### Charm class hierarchy

All framework charms ultimately inherit from `ops.CharmBase`:

```
ops.CharmBase
  └─ PaasCharm          (src/paas_charm/charm.py) — abstract; handles all integrations
       ├─ GunicornBase  (src/paas_charm/_gunicorn/charm.py) — WSGI frameworks (Flask, Django, FastAPI)
       │    ├─ flask.Charm
       │    ├─ django.Charm
       │    └─ fastapi.Charm
       └─ (direct subclasses for non-WSGI frameworks)
            ├─ go.Charm
            ├─ expressjs.Charm
            └─ springboot.Charm
```

`PaasCharm.__init__` sets up *all* possible integrations (databases, Redis, Valkey, S3, SAML, RabbitMQ, tracing, SMTP, OpenFGA, HTTP proxy, ingress, OAuth, observability) by introspecting the charm metadata `requires` dict. Optional integrations are imported with `try/except ImportError` to preserve backwards compatibility when charm libraries have not been fetched.

### Framework config models

Each framework defines a `FrameworkConfig(pydantic.BaseModel)` subclass (e.g. `DjangoConfig`, `GoConfig`). Fields use `alias=` to match the Juju charm config key name (e.g. `Field(alias="django-debug")`). The `FrameworkConfig.secret_key_id` validator transparently resolves `<config>-id` secret references to their plaintext values.

### CharmState

`CharmState` (`src/paas_charm/charm_state.py`) is a plain data class constructed by `CharmState.from_charm(...)` in each event handler. It aggregates:
- `framework_config` — typed framework config model instance
- `user_defined_config` — extra `app-` prefixed charm config keys passed straight through to the workload as env vars
- `integrations` — an `IntegrationsState` holding typed relation data objects for every active integration
- `secret_key` — managed secret key stored in a peer relation

### WorkloadConfig and App

`WorkloadConfig` (dataclass in `src/paas_charm/app.py`) describes the in-container workload: paths, ports, service names, metrics endpoints. Each framework's `Charm._workload_config` property constructs one.

`App` manages the Pebble layer, environment variable composition, and `restart()` for the workload container. Integration data is converted to env vars by the various `*RelationData` models.

### paas-config.yaml

A `paas-config.yaml` file **inside the OCI image** (read via `read_paas_config()`) configures framework-level settings such as `framework_logging_format` and custom Prometheus scrape configs. This is distinct from Juju charm config.

### COS (observability)

Each framework directory has a `cos/` subdirectory with Grafana dashboards, Loki rules, and Prometheus alert rules. `PaasCharm.build_cos_dir()` merges the base `src/paas_charm/templates/` COS files with the framework-specific ones.

## Key Conventions

### Docstrings and copyright

Every Python file must start with a copyright header:
```python
# Copyright <year> Canonical Ltd.
# See LICENSE file for licensing details.
```
All public symbols require Google-style docstrings. The flake8-docstrings-complete plugin enforces that `Attrs:`, `Args:`, `Returns:`, and `Raises:` sections are present and complete.

### Code style

- **Line length**: 99 characters (black + flake8 both configured to 99)
- **Formatter**: `black --target-version py310` + `isort` (profile `black`)
- **Type annotations**: required on all functions (`mypy --disallow-untyped-defs`); `disallow_untyped_defs = false` only in tests

### Test structure

- Unit tests use `ops.testing.Harness`. Each framework subdirectory has a `constants.py` with `DEFAULT_LAYER` and relation data fixtures used across tests.
- The `[testenv:unit]` tox env requires `charmcraft fetch-libs` to be run first (handled automatically by tox via the `commands` section which installs charmcraft and fetches libs).
- Coverage minimum is **90%** (`fail_under = 90` in `pyproject.toml`).

### Adding a new integration

New integrations go in `src/paas_charm/` as a dedicated module (e.g. `rabbitmq.py`). The import must be wrapped in `try/except ImportError` with a `logger.warning(...)` pointing to the `charmcraft fetch-lib` command. Wire it up in `PaasCharm.__init__` using the same pattern as existing integrations (check `requires` metadata, conditionally create the requirer, register observers).

### Adding a new framework

1. Create `src/paas_charm/<framework>/charm.py` with a `FrameworkConfig` model and a `Charm` class inheriting from `PaasCharm` (or `GunicornBase` for WSGI).
2. Add a `cos/` subdirectory for observability assets.
3. Add example charm under `examples/<framework>/`.
4. Add unit tests under `tests/unit/<framework>/` with a `constants.py`, `conftest.py`, and test files mirroring other frameworks.
5. Coordinate a matching Rockcraft extension and Charmcraft extension (in separate repos) before merging.

### Commits

All commits must be signed (GPG) and include a CLA sign-off line at the end. PRs are squash-merged onto `main`. Update `docs/changelog.md` for any feature, fix, or significant change.
